# 🏠 高端租屋代管平台｜角色權限與加入流程規格 (roles.md)

## 1. 角色定義概述

本系統採用雙層權限架構：
- **系統角色 (System Role)**: 決定用戶在全系統層級的基礎權限。
- **組織角色 (Member Role)**: 決定用戶在特定組織 (Organization) 內的管理權限。

### 1.1 系統角色 (System Role)
對應資料庫 `User` 資料表的 `systemRole` 欄位。

| 角色名稱 | 列舉值 (Enum) | 跨組織存取 | 核心職責 | 訂閱主體 |
| :--- | :--- | :--- | :--- | :--- |
| **平台管理員** | `ADMIN` | 是 | 系統維運、邀請 Landlord/Manager 並設定方案、全平台用戶管理 (停權/刪除)。不做租賃事務。 | 否 |
| **房東 (自主)** | `LANDLORD` | 否 | 擁有房源資產，自主管理物業，負責財務與合約。 | 是 (Free/Starter) |
| **代管業者** | `MANAGER` | 是 | 負責多個房東的房源維護與維運。**可加入多個不同房東的組織。** | 是 (Starter/Pro) |
| **房客** | `TENANT` | 否 | 最終使用者，負責度數回報、繳費、報修。 | 否 |
| **訪客** | `VISITOR` | 否 | 未登入用戶。僅能查看被標記為「公開 (Public)」或「授權顯示」之房源摘要資訊。 | 否 |

### 1.2 組織成員角色 (Member Role)
對應資料庫 `UserOrganization` 資料表的 `memberRole` 欄位。

- **OWNER**: 組織擁有者（通常為 Landlord），擁有該組織的所有管理權限。
- **MANAGER**: 組織管理員（通常為 Manager），擁有受限的管理權限，無法刪除組織或變更擁有者。

## 3. 註冊與加入流程

### 3.1 房東 (Landlord) 與 代管 (Manager) 加入流程
1. **觸發**: 由 `ADMIN` 於管理後台發送邀請郵件，並預設其訂閱方案 (Plan)。
2. **註冊**: 受邀者點擊連結，填寫基本資料完成註冊。
3. **初始化**:
   - 系統自動建立專屬 `Organization`。
   - 將該用戶設為該組織的 `OWNER`。
   - 附加指定的 `SubscriptionPlan` 於該組織。

### 3.2 組織內協作流程 (Manager 協助 Landlord)
1. **觸發**: 由 `LANDLORD` 在組織管理頁面生成邀請連結。
2. **註冊**: 受邀人員登入或註冊。
3. **繫結**: 系統將用戶加入該 `Organization`，賦予輔助管理權限。**一位 Manager 可同時接受多位 Landlord 邀請。**

### 3.3 房客 (Tenant) 流程
1. **觸發**: 房東或代管人員針對特定「房源 (Property)」生成邀請碼。
2. **註冊**: 房客進入註冊頁面，輸入邀請碼。
3. **綁定**: 
   - 系統驗證邀請碼有效性。
   - 建立房客帳號 (`systemRole: TENANT`)。
   - **自動建立租約 (Contract)** 關聯，將房客與房源、組織連結。
4. **限制**: **嚴格禁止**無邀請碼的自主註冊，以確保高端代管的封閉性與資料正確性。

## 4. 停權行為與影響矩陣 (Suspension Policy)

當用戶狀態 (`User.status`) 被變更為 `SUSPENDED` 時，其關聯資料與存取權限受以下限制：

| 被停權角色 | 對資產的影響 | 對協作成員的影響 | 對租務流程的影響 |
| :--- | :--- | :--- | :--- |
| **ADMIN** | 無立即物理影響，僅該帳號無法進入管理後台。 | 其他 Admin 仍可運作。 | 無影響。 |
| **LANDLORD** | 所屬組織所有房源自動隱藏 (Private)，訪客無法搜尋。 | 該組織下的 Manager 轉為「唯讀」。無法新增任何資料。 | 租約與帳單流程凍結，僅限讀取歷史資料。 |
| **MANAGER** | 負責房源的 `managerId` 自動解除關聯或掛起。 | 系統通知 Landlord 該代管已被停權，需重新指派。 | 該 Manager 經手的待辦事項 (如報修初審) 全數回歸給 Landlord。 |
| **TENANT** | 無法登入 App 進行任何操作。 | Landlord/Manager 收到該房客被停權之稽核通知。 | 禁止發起報修、線上繳費。已產生的帳單仍處於有效狀態。 |

## 5. 註解與技術限制 (Developer Notes)

- **唯一性限制**: 一個 `User` 帳號在目前的實作中，只能對應一個主要 `systemRole`。
- **組織 (Organization) 的必要性**: 它是權限隔離與方案計費的最小單位，不可輕易刪除。
- **多組織存取 (Cross-Org Access)**: 
  - `Manager` 的存取需透過 `UserOrganization` 表中多條紀錄來達成。這支持了代管同時管理多個不同房東資產的需求。
- **邀請碼安全性**: 邀請碼內嵌 `PropertyID` 與 `OrganizationID` 的加密資訊，並設有有效期 (`INVITATION_EXP_DAYS`)。
- **自動化邏輯**: 房客註冊完成的瞬間，必須確保 `Contract` 狀態從 `PENDING` 轉為 `ACTIVE` 或直接建立有效合約，不可出現孤立房客。
