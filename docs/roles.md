# 🏠 高端租屋代管平台｜角色權限與加入流程規格 (roles.md)

## 1. 角色定義概述

本系統採用雙層權限架構：
- **系統角色 (System Role)**: 決定用戶在全系統層級的基礎權限。
- **組織角色 (Member Role)**: 決定用戶在特定組織 (Organization) 內的管理權限。

### 1.1 系統角色 (System Role)
對應資料庫 `User` 資料表的 `systemRole` 欄位。

| 角色名稱 | 列舉值 (Enum) | 跨組織存取 | 核心職責 | 訂閱主體 |
| :--- | :--- | :--- | :--- | :--- |
| **平台管理員** | `ADMIN` | 是 | 系統維運、邀請 Landlord/Manager 並設定方案、全平台用戶管理 (停權/刪除)。不做租賃事務。 | 否 |
| **房東 (自主)** | `LANDLORD` | 否 | 擁有房源資產，自主管理物業，負責財務與合約。 | 是 (附屬於其 Organization) |
| **代管業者** | `MANAGER` | 是 | 負責多個房東的房源維護與維運。**可加入多個不同房東的組織。** | 是 (附屬於自營 Organization) |
| **房客** | `TENANT` | 否 | 最終使用者，負責度數回報、繳費、報修。 | 否 |
| **訪客** | `VISITOR` | 否 | 未登入用戶。僅能查看被標記為「公開 (Public)」或「授權顯示」之房源摘要資訊。 | 否 |

### 1.2 組織成員角色 (Member Role)
對應資料庫 `UserOrganization` 資料表的 `memberRole` 欄位。

- **OWNER**: 組織擁有者（通常為 Landlord），擁有該組織的所有管理權限。
- **MANAGER**: 組織管理員（通常為 Manager），擁有受限的管理權限，無法刪除組織或變更擁有者。

## 3. 註冊與加入流程 (Genesis Entry)

### 3.1 組織 (Organization) 的建立與權屬
1. **建立者**: **ADMIN (平台管理員)** 是組織產生的發起者。
2. **建立契機**:
   - 當 Admin 發送「房東型」或「專業代管型」邀請時，需預指派一個 `SubscriptionPlan`。
   - 用戶點擊連結註冊時，系統會在註冊 Transaction 中**自動初始化**該組織。
3. **初始化邏輯**:
   - 系統建立專屬 `Organization`。
   - 該註冊用戶成為該組織的 `OWNER` (擁有者)。
   - 訂閱方案（如 Starter/Pro）直接賦予該組織。

### 3.2 組織內協作流程 (Manager 協助 Landlord)
1. **觸發**: 由 `LANDLORD` 在組織管理頁面生成邀請連結。
2. **註冊**: 受邀人員登入或註冊。
3. **繫結**: 系統將用戶加入該 `Organization`，賦予輔助管理權限。**一位 Manager 可同時接受多位 Landlord 邀請。**

### 3.3 專業代管公司 (Professional Agency)
- 與一般受僱 Manager 不同，專業代管公司擁有「自己的組織」。
- 建立方式同 3.1 節，由 Admin 邀請後建立獨立組織。
- 代管公司的組織具有自己的 Pro 級別訂閱。
- **跨組織代管**: 代管人可在「自營組織」中建立房源（消耗自己的 PRO 額度），或被邀請進入「房東組織」中管理房源（消耗房東的額度）。

### 3.3 房客 (Tenant) 流程
1. **觸發**: 房東或代管人員針對特定「房源 (Property)」生成邀請碼。
2. **註冊**: 房客進入註冊頁面，輸入邀請碼。
3. **綁定**: 
   - 系統驗證邀請碼有效性。
   - 建立房客帳號 (`systemRole: TENANT`)。
   - **自動建立租約 (Contract)** 關聯，將房客與房源、組織連結。
4. **限制**: **嚴格禁止**無邀請碼的自主註冊，以確保高端代管的封閉性與資料正確性。

## 4. 停權行為與影響矩陣 (Suspension Policy)

當用戶狀態 (`User.status`) 被變更為 `SUSPENDED` 時，其關聯資料與存取權限受以下限制：

| 被停權角色 | 對資產的影響 | 對協作成員的影響 | 對租務流程的影響 |
| :--- | :--- | :--- | :--- |
| **ADMIN** | 無立即物理影響，僅該帳號無法進入管理後台。 | 其他 Admin 仍可運作。 | 無影響。 |
| **LANDLORD** | 所屬組織所有房源自動隱藏 (Private)，訪客無法搜尋。 | 該組織下的 Manager 轉為「唯讀」。無法新增任何資料。 | 租約與帳單流程凍結，僅限讀取歷史資料。 |
| **MANAGER** | 負責房源的 `managerId` 自動解除關聯或掛起。 | 系統通知 Landlord 該代管已被停權，需重新指派。 | 該 Manager 經手的待辦事項 (如報修初審) 全數回歸給 Landlord。 |
| **TENANT** | 無法登入 App 進行任何操作。 | Landlord/Manager 收到該房客被停權之稽核通知。 | 禁止發起報修、線上繳費。已產生的帳單仍處於有效狀態。 |

## 5. 註解與技術限制 (Developer Notes)

- **唯一性限制**: 一個 `User` 帳號在目前的實作中，只能對應一個主要 `systemRole`。
- **組織 (Organization) 的必要性**: 它是權限隔離與方案計費的最小單位，不可輕易刪除。
- **多組織存取 (Cross-Org Access)**: 
  - `Manager` 的存取需透過 `UserOrganization` 表中多條紀錄來達成。這支持了代管同時管理多個不同房東資產的需求。
- **邀請碼安全性**: 邀請碼內嵌 `PropertyID` 與 `OrganizationID` 的加密資訊，並設有有效期 (`INVITATION_EXP_DAYS`)。
- **自動化邏輯**: 房客註冊完成的瞬間，必須確保 `Contract` 狀態從 `PENDING` 轉為 `ACTIVE` 或直接建立有效合約，不可出現孤立房客。

## 6. 訂閱與級別判定設計 (Subscription Logic)

### 6.1 組織中心制 (Organization-Centric)
訂閱方案與額度計費是以 **Organization** 為主體。用戶的存取權限與可管理房源數，取決於當前所切換的「工作區（Organization）」之方案。

### 6.2 分場景判定規則

| 情境 | 訂閱判定主體 | 費用承擔 | 邏輯說明 |
| :--- | :--- | :--- | :--- |
| **房東邀請代管** | 房東的組織 | 房東 | 代管人僅為組織之協作成員，共享該組織之方案額度。 |
| **代管接受授權** | 房源所屬組織 | 視歸屬而定 | 若房源隸屬於房東組織，使用房東方案；若房源已轉移至代管公司組織，使用代管方案。 |
| **自營與協作並行** | 當前工作區 | 各別擁有者 | 同一用戶在 A 組織 (Owner) 享有 Pro，在 B 組織 (Member) 僅享有 Free，權限不互通。 |

## 7. Nexus Index 導航架構與工作區映射

為確保管理中心 (Nexus Pulse) 的一致性，定義以下導航與顯示邏輯：

### 6.1 登入角色設計導向 (Design Orientation)

| 登入角色 (Role) | 索引起始點 (Root) | 設計導向 (Perspective) |
| :--- | :--- | :--- |
| **全域管理員 (ADMIN)** | **組織 (Organization)** | 「上帝視角」：負責全平台生態監控、方案稽核。 |
| **房東 (LANDLORD)** | **個人組織 / 旗下房源** | 「資產視角」：專注於旗下資產之營收與決策。 |
| **物業經理 (MANAGER)** | **所屬房東 / 指派房源** | 「執行視角」：專注於房源維護與租客溝通。 |

### 6.2 深度統計與成員提取邏輯 (Nexus Index Logic)

管理中心工作區遵循「深度向上彙總」與「成員穿透顯示」原則：

| 選取物件 (Node) | 實體數 (Stats) | 授權人 (Stats) | 成員清單 (Users Tab) | 關聯實體 (Grid) |
| :--- | :--- | :--- | :--- | :--- |
| **組織 (Org)** | 全組織下轄**房源總數** | 組織下轄**房東總數** | 組織擁有者 + 旗下**所有房東** + 全組織境內**房客** | 房東 (Landlord) 列表 |
| **房東 (Landlord)** | 該房東名下**房源總數** | 1 (房東本人) | 房東本人 + 其名下房源之**所有房客** | 房源 (Property) 列表 |
| **房源 (Property)** | 1 (物件本身) | 1 (擁有者) | 負責房東 + 物業經理 + **現任租客** | 租客與經理資訊卡 |

### 6.3 實作技術規則 (Developer Implementation Rules)

1. **數據統計 (Stats)**：
   - 為解決 Lazy Loading 導致的深度缺失，API 必須在 metadata 中預計回傳 `propertiesCount` 與 `landlordsCount`。
   - 前端優先使用預計值，若無則降級為遞迴計算。
2. **成員提取 (Users)**：
   - **穿透顯示**：成員清單不應受限於直接子代。
   - 在前端緩取機制中，若父節點已被點擊，應將其下轄具備 User 類型的節點（Landlord, Tenant）全數扁平化顯示。
